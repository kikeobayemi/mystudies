Secure Software Development - Learning Outcomes

  • Identify and manage security risks as part of a software development project.

  • Critically analyse development problems and determine appropriate methodologies, tools and techniques (including program design and development) to solve them.

  • Design and develop/adapt computer programs and to produce a solution that meets the design brief and critically evaluate solutions that are produced.

  • Systematically develop and implement the skills required to be effective member of a development team in a virtual professional environment, adopting real-life perspectives on team roles and organisation.

    Learning Outcomes

    • Decompose an activity into a set of tasks which can be represented using a flow chart from the UML portfolio of modelling techniques.

I have no prior knowledge of this topic however I was happy to learn about the security challenges of web applications and to increase awareness on how to create more secure software which is basically the focus of the OWASP organisation. I am really looking forward to developing skills on how to create secure solutions throughout this module.

    Exploring Programming Language Concepts

    Learning Outcomes
    • Explain how and when to use regular expressions in solutions.
    • Describe how and when to use recursion in solutions.
    • Discuss the security implications of both approaches.

    Reflection

    We had two programming exercises this week: Recursion and Regex. The key learning points here for me with relation to secure coding are;

    Recursion: to understand how to execute recursive functions securely. Recursion can be prone to overflow errors when not properly written (Fischer, 2014)

    Regex: to learn how to create strict patterns (such as passwords) and also to know the characters (or combination of characters) that can subject your code to a ReDoS attack. I found the tutorial in Jaiswal, S. (2020) Python Regular Expression Tutorial very helpful and I was able to write my regex for the UK postcodes whilst making sure to avoid evil regex patterns as shown in Weidman A. (n.d) Regular Expression – Denial of Service. I also discovered efficiency can be brought into writing regular expressions; on comparing my regex with that of my team mates, I discovered my regex was shorter in length and it still worked fine.

    On the summative assessment, I shared an outline containing all the relevant sections for the proposal with my team mates and encouraged everyone to start posting a rough draft of the sections assigned so others could review. This gave us a clear direction and spurred everyone including myself to action.

Other team mates focused on tools and implementation of the API. Basically, we all played to our strengths and took on tasks that we felt confident delivering.

    Future trends in Secure Software Development

    Learning Outcomes

  • Give examples of fog computing, IoT and Cyber Physical System components and solutions.
  • Describe some of the security issues with the above-mentioned systems.
  • Recommend emerging technologies and solutions to investigate.

    • Reflection

    We had an interesting discussion this week on the Torvalds-Tanenbaum debate in which I learnt about monolithic and microkernel operating systems (OS). I also read extensively about the history of Linux as a monolithic OS, its usage across all kinds of computers and security issues with respect to its huge code base. I came across an exciting term “bug density” in my research which relates to the amount of bugs per line of code.

    From a security point of view, microkernels appear to be safer because they have fewer lines of executable kernel code (Bissyande, 2013). However, reports show that monoliths have a huge level of dominance in the market (Computerworld, 2008; Statista, 2021). The question on my mind then is “Are operating systems developers not giving a higher priority to Security?” I spent more time conducting research on this topic than any other in the module. As we speak, I am still looking for conclusive answers to my question.

    As for our group project, we completed all the outstanding functionalities in the API. I did some functionality testing and developed some parts of the documentation. In comparison to the proposal we submitted in week 6, we did not implement the privileges as designed. It was a bit challenging and there wasn’t much time left. In the end, we created just two types of privileges as opposed to three which was originally planned. I think we underestimated the time involved in carrying out this project coupled with the fact that all three of us were taking double modules. Next time, I would start preparing early to avoid rushing and leaving out some bits on an assignment.
