Secure Software Development - Learning Outcomes

  • Identify and manage security risks as part of a software development project.

  • Critically analyse development problems and determine appropriate methodologies, tools and techniques (including program design and development) to solve them.

  • Design and develop/adapt computer programs and to produce a solution that meets the design brief and critically evaluate solutions that are produced.

  • Systematically develop and implement the skills required to be effective member of a development team in a virtual professional environment, adopting real-life perspectives on team roles and organisation.

    • Introduction to Secure Software Development

    Learning Outcomes

    • Identify relevant academic literature which discusses the ways in which secure software may be developed using an agile process.
    • Decompose an activity into a set of tasks which can be represented using a flow chart from the UML portfolio of modelling techniques.
    • Describe the importance of creating a culture of risk awareness in an organisation, in addition to the creation of secure software.

    Reflection

    In the introductory week, I was drafted into Team 3 for the module projects which will be submitted in Week 6 and 11 respectively. My team mates and I created a team contract for the group and sent it to the tutor. I find the idea of working in a group quite interesting especially because my team mates have a wide range of skills and everyone communicates effectively so far.

    For the week 1 activity, I created a collaborative discussion post and flowchart on the OWASP Top 10 coding weaknesses. I have no prior knowledge of this topic however I was happy to learn about the security challenges of web applications and to increase awareness on how to create more secure software which is basically the focus of the OWASP organisation. I am really looking forward to developing skills on how to create secure solutions throughout this module.

    Exploring Programming Language Concepts

    Learning Outcomes
    • Explain how and when to use regular expressions in solutions.
    • Describe how and when to use recursion in solutions.
    • Discuss the security implications of both approaches.

    Reflection

    We had two programming exercises this week: Recursion and Regex. The key learning points here for me with relation to secure coding are;

    Recursion: to understand how to execute recursive functions securely. Recursion can be prone to overflow errors when not properly written (Fischer, 2014)

    Regex: to learn how to create strict patterns (such as passwords) and also to know the characters (or combination of characters) that can subject your code to a ReDoS attack. I found the tutorial in Jaiswal, S. (2020) Python Regular Expression Tutorial very helpful and I was able to write my regex for the UK postcodes whilst making sure to avoid evil regex patterns as shown in Weidman A. (n.d) Regular Expression – Denial of Service. I also discovered efficiency can be brought into writing regular expressions; on comparing my regex with that of my team mates, I discovered my regex was shorter in length and it still worked fine.

    On the summative assessment, I shared an outline containing all the relevant sections for the proposal with my team mates and encouraged everyone to start posting a rough draft of the sections assigned so others could review. This gave us a clear direction and spurred everyone including myself to action.

    An Introduction to Testing

    Learning Outcomes

    • Describe the key terms which are associated with testing software for security.
    • Prepare a list of questions to ask when designing a test plan for secure software.
    • Design software tests through understanding the range of ways which the security of software can be breached.

    Reflection

    This week we discussed Introduction to testing and I began to get a better appreciation of testing other aspects of a code besides functionality. Before this module, I only focused on testing if the code works; I thought as long as a code does its job, then it is complete.

    On the formative activities, I did not really grasp the concept of Equivalence testing with respect to testing either the functionality, usability, performance or security of a code but I enjoyed working on the task Exploring the relevance of cyclomatic complexity today. It is worth noting that a program with high complexity value has a high probability for errors (Hamilton, 2021)

    For the summative assessment, I started my research on the large hadron collider and some of the security challenges of research institutions such as CERN. Other team mates focused on tools and implementation of the API. Basically, we all played to our strengths and took on tasks that we felt confident delivering.
