    Fundamentals of Object-Oriented Design

    Learning Outcomes
  • Identify the appropriate objects within a system.
  • Develop an object-oriented design for a system.
  • Correctly apply composition and inheritance where appropriate.

    • Summary/Reflection

    Object-oriented design involves implementing a software in terms of objects and a set of methods that define meaningful operations on the objects (Olagunju & Akpan, 2015). Object-oriented techniques are widely used in developing information systems and in the computing field at large (Paetau, 1997).

    The second lecturecast of the module titled “Fundamentals of Object-Oriented Design (OOD)” provided some key insights into the topic. The main points are summarized below:

  • An Object is a data structure with 2 main components: data (attributes) and procedures (methods or functions)
  • Four(4) fundamental concepts used in OOD: Abstraction which involves hiding inbuilt properties of our object and only showing what the user needs. Encapsulation:the processing of bundling data and methods together. Inheritance which allows an object to use attributes and functions from another object and Polymorphism which allows an object to take multiple forms
  • Advantages of OOD: It helps to model real world effectively.It supports system flexibility and reusability.
  • Relationships in OOD:

    • Association: is a relationship between two objects without any particular relationship.
    Composition: A parent and child relationship whereby the child cannot exist without the parent. For example, when a bag is destroyed, all the internal pockets are destroyed. Composition depicts the ‘is part of’ relationship.
    Aggregation: A parent and child relationship whereby the child can still exist without the parent. For example, wheels can still exist outside of a car. Aggregation depicts the ‘has a’ relationship

    Understanding UML

    Learning Outcomes

  • Use UML to develop an object-oriented system design.
  • Identify and use the correct elements of UML to design a system.

    • Summary/Reflections

    I started learning about UML diagrams during our practical activity in week 3 where I modelled an information system – Online Supermarket. This made the week 5 lecturecast: Understanding UML very easy to follow. The main points are summarized below:

  • UML diagrams are a standardized way of communicating information system designs.
  • Use case diagrams are mainly used for capturing requirements, planning iteration of development and validating a system.
  • Class diagrams are the most commonly used UML diagrams. They are used to model objects and the relationships between them. They also show data (attributes) and functions that are performed on them.
    • Note: the focus for this week is class diagrams

    This week, I continued with the week 5 lecturecast: Understanding UML but this time with focus on other types of UML diagrams. UML diagrams are divided into 2 broad categories: Structure Diagram and Behavioural Diagrams. Object and Class diagrams covered in previous weeks fall under the structure category.Some behavioural diagrams that we looked at are:

  • Sequence diagram: describes how objects interact with each other and in what order (lucidchart, 2021)
  • Activity diagram: describes the workflow in the system being modelled (creately, 2021)
  • State diagram: is also a behavioural diagram that describes behaviour of objects according to the state that they are in.(creately, 2021)

    • Database Design

    Learning Outcomes

  • Develop the knowledge and skills to apply database design principles.
  • Design a database that has been correctly normalised .

    • Summary/Reflection

    “Data is the new oil” – Clive Humbly, 2006
    In this week’s lecturecast, we looked at relational databases and SQL, the technology that underpins them with a focus on normalization.
    Database is a structured approach to data storage. It is made up of 2 key ingredients: (i) The data itself & (ii) Data dictionary. It consists of 3 components:

  • Entities: they represent objects
  • Attributes: the elements that make up the entity
  • Relationships: how two entities might be joined together (for example composition, inheritance)
    • Relational databases
    A table with rows (also referred to as tuples) and columns (fields or attributes) is known as a relation.

    Normalisation
    Normalisation aims to prevent data anomalies such as insertion, deletion, and modification anomaly. Normalisation involves taking a table from 1st Normal Form to Second Normal Form and then to Third Normal Form. Keys are used to achieve normalisation and maintain the core elements of relational database.

  • Primary Key: is unique for each record
  • Composite key: using two fields to create our unique identifier: e.g street name and postcode columns
  • Foreign key references another column in another table. e.g column in Property Issues table will make reference to the exact property in Properties table.
